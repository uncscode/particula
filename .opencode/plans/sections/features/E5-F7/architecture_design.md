# Architecture Design

### High-Level Design

Issue #1362 implements the deterministic oracle/observation boundary as two
co-located private test modules. `_coagulation_validation_support.py` owns
literal row metadata, fp64 fixtures, and host-only expected values; the
discoverable `coagulation_validation_test.py` keeps Warp imports and private
device-helper probes lazy. The latter observes properties, pair rates, and the
selector majorant on Warp CPU, while the former never uses Warp aggregate logic
as an expected-value source. Deferred masks are resolved and rejected on the
host before any execution observation.

Issues #1363 and #1364 share a private lazy-Warp public-step layer in
`_coagulation_public_step_support.py`. It materializes `ParticleData` with
species-resolved mass, active-slot concentration, charge, inactive sentinels,
and optional P3 volume/concentration overrides; conversion creates caller-owned
Warp state. Snapshots synchronize mutable particle, collision-pair/count, and
RNG sidecars around each call. The invariant layer validates accepted pair
prefixes and resulting bookkeeping rather than a stochastic pair sequence.
Runtime device selection stays lazy: Warp CPU is required when installed and
CUDA is additive.

E5-F7 is a validation layer around the stable low-level coagulation entry point;
it does not add a production execution path. A canonical case table binds each
supported mechanism mask to independently evaluated deterministic expectations,
physical invariants, stochastic expectations, and applicable input sidecars.

```text
Canonical mechanism/case table
  +-> independent CPU API or direct NumPy pair/property oracle
  |     -> deterministic rate and majorant assertions
  +-> fresh fixed-shape WarpParticleData fixture
        -> coagulation_step_gpu(configuration, buffers, rng_states)
             -> required Warp CPU run
             -> optional CUDA run when available
        -> per-run legal-pair + ownership + conservation assertions
        -> repeated-seed aggregate statistics
             -> documented confidence/sigma bound
  -> generated human-readable support/evidence matrix
```

The deterministic and stochastic checks remain separate. A stochastic tolerance
may never relax mass, charge, inactive-slot, capacity, or ownership assertions,
which run on every sample. Expected totals are not generated by the Warp helper
under test.

### Data / API / Workflow Changes

- **Data Model:** No production schema changes. Test cases use explicit fp64
  NumPy inputs and existing `WarpParticleData` fixed shapes. Small frozen test
  case records may describe mechanism masks, required sidecars, tolerances, and
  expected aggregate rates.
- **API Surface:** No new public package API is planned. Tests call the shipped
  `coagulation_step_gpu()` configuration contract from E5-F1 through E5-F6.
- **Workflow Hooks:** Deterministic probes and P2 public-entry tests use `warp`
  and `gpu_parity`. The P3 device matrix additionally uses `stochastic` and
  statically parameterizes CPU/CUDA; only CUDA parameters carry `cuda` and
  runtime-skip through the unchanged shared availability helper. Normal
  collection stays Warp-free, while selected device observations import Warp
  lazily.
- **Evidence Boundary:** The published table reports only rows implemented and
  approved by E5-F3 through E5-F6. Missing or unsupported rows are documented as
  unsupported, not silently skipped as successful.

### Security & Compliance

No external services, credentials, or new permissions are introduced. Test
fixtures must remain deterministic and bounded in memory/runtime. Device
probing uses the shared warning-scoped CUDA helper; it must not swallow unrelated
warnings. Failure diagnostics may report dimensions, mechanism names, seeds,
and numeric deltas but must not mutate caller state to obtain them.
