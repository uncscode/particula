# Platform Router Setup Agent - Usage Guide

## Overview

The `setup-platform-dev` agent is an interactive wizard that helps developers configure fork/upstream platform routing in their `.env` file. It generates a focused `.env` snippet that users can paste into their existing environment file.

This agent handles:
1. Platform selection (GitHub vs GitLab)
2. Workflow type detection (single-repo vs fork/upstream)
3. Issue routing configuration (`ADW_TARGET_REPO`)
4. Repository URLs and token setup
5. Operation permissions (simple defaults or advanced customization)
6. Branch hierarchy configuration
7. PR scanning enablement

## When to Use

| Scenario | Use This Agent |
|----------|----------------|
| First-time fork/upstream routing setup | Yes |
| Configuring where issues come from | Yes |
| Setting up dual-token authentication | Yes |
| Enabling dual-scan mode (`both`) | Yes |
| Customizing operation permissions | Yes |
| Configuring branch hierarchy | Yes |
| Enabling PR scanning | Yes |
| Full environment setup from scratch | No - use `adw setup env` |
| Setting up ADW templates | No - use `setup-adw` |

## Conversation Flow

The agent guides users through a step-by-step wizard:

### Step 1: Platform Selection
```
Which platform does your repository use?
1. GitHub (github.com or GitHub Enterprise)
2. GitLab (gitlab.com or self-hosted)
```

### Step 2: Workflow Type
```
How is your repository set up?
1. Single repository - I have direct access to one repo for everything
2. Fork/upstream - I work on a fork and the main project is upstream
```

### Step 3: Issue Routing (Fork/Upstream only)
```
Where should ADW fetch issues from?
1. Fork only (ADW_TARGET_REPO=fork)
2. Upstream only (ADW_TARGET_REPO=upstream) [RECOMMENDED]
3. Both/Dual-scan (ADW_TARGET_REPO=both)
```

### Step 4: Repository URLs
Collects fork URL and optionally upstream URL.

### Step 5: Token Configuration
Collects tokens with option to reuse fork token for upstream.

### Step 6: Operation Permissions
```
1. Use defaults (recommended)
2. Customize operations (advanced)
```

### Step 7: Branch Hierarchy
```
1. Disable hierarchy (branches target main)
2. Enable hierarchy (use develop as integration branch)
```

### Step 8: PR Scanning
```
1. Disable (default) - Only scan issues
2. Enable - Scan both issues and PRs
```

### Step 9: Validation (Optional)

The agent can test the configuration before generating the snippet:

```
Would you like me to test your configuration?
1. Yes - verify tokens and access
2. No - generate snippet directly
```

If validation is chosen, the agent:
1. Tests `rate-limit` to verify fork token is valid
2. Tests `rate-limit --prefer-scope upstream` if upstream is configured
3. Optionally fetches an issue to confirm routing works

**Example validation output:**

## Output Format

The agent generates a `.env` snippet with inline documentation:

```bash
# ==============================================================================
# Platform Routing Configuration
# Generated by: setup-platform-dev agent
# ==============================================================================

# Target for issue/label operations: fork, upstream, or both
ADW_TARGET_REPO=upstream

# Your fork (where branches and PRs are created)
GITHUB_REPO_URL=https://github.com/yourusername/project

# Upstream repository (where issues live)
GITHUB_UPSTREAM_URL=https://github.com/organization/project

# Authentication
GITHUB_PAT=ghp_your_fork_token_here
GITHUB_UPSTREAM_PAT=ghp_your_upstream_token_here

# Branch Hierarchy
ADW_BRANCH_HIERARCHY=true
ADW_DEV_BRANCH=develop

# PR Scanning
ADW_ENABLE_PR_SCANNING=true
```

## Common Scenarios

### Scenario 1: Single Repository

For users with direct access to one repository:

```bash
ADW_TARGET_REPO=fork
GITHUB_REPO_URL=https://github.com/owner/project
GITHUB_PAT=ghp_token
ADW_BRANCH_HIERARCHY=false
ADW_ENABLE_PR_SCANNING=false
```

### Scenario 2: Fork Contributor

For open source contributors working on a fork:

```bash
ADW_TARGET_REPO=upstream
GITHUB_REPO_URL=https://github.com/yourusername/fork
GITHUB_UPSTREAM_URL=https://github.com/organization/project
GITHUB_PAT=ghp_fork_token
GITHUB_UPSTREAM_PAT=ghp_upstream_token
ADW_BRANCH_HIERARCHY=true
ADW_DEV_BRANCH=develop
ADW_ENABLE_PR_SCANNING=true
```

### Scenario 3: Dual-Scan Maintainer

For maintainers handling issues from both fork and upstream:

```bash
ADW_TARGET_REPO=both
GITHUB_REPO_URL=https://github.com/maintainer/fork
GITHUB_UPSTREAM_URL=https://github.com/organization/project
GITHUB_PAT=ghp_fork_token
GITHUB_UPSTREAM_PAT=ghp_upstream_token
GITHUB_FORK_OPS=issue:read,issue:write,...
GITHUB_UPSTREAM_OPS=issue:read,issue:write,...
ADW_BRANCH_HIERARCHY=true
ADW_DEV_BRANCH=develop
ADW_ENABLE_PR_SCANNING=true
```

## Environment Variables Reference

### ADW_TARGET_REPO

| Value | Behavior |
|-------|----------|
| `fork` | All operations target the fork (default) |
| `upstream` | Issues/labels from upstream, PRs on fork |
| `both` | Dual-scan: fork first, then upstream, merge results |

### Repository URLs

| Variable | Platform | Description |
|----------|----------|-------------|
| `GITHUB_REPO_URL` | GitHub | Fork repository URL |
| `GITHUB_UPSTREAM_URL` | GitHub | Upstream repository URL |
| `GITLAB_REPO_URL` | GitLab | Fork project URL |
| `GITLAB_UPSTREAM_URL` | GitLab | Upstream project URL |

### Tokens

| Variable | Platform | Description |
|----------|----------|-------------|
| `GITHUB_PAT` | GitHub | Fork token (scopes: repo, workflow) |
| `GITHUB_UPSTREAM_PAT` | GitHub | Upstream token |
| `GITLAB_TOKEN` | GitLab | Fork token (scopes: api, read_repository, write_repository) |
| `GITLAB_UPSTREAM_TOKEN` | GitLab | Upstream token |

### Operation Permissions

| Variable | Description |
|----------|-------------|
| `GITHUB_FORK_OPS` / `GITLAB_FORK_OPS` | Operations for fork token |
| `GITHUB_UPSTREAM_OPS` / `GITLAB_UPSTREAM_OPS` | Operations for upstream token |

**Available Operations:**
- `issue:read`, `issue:write`, `issue:comment`
- `label:read`, `label:write`
- `pr:read`, `pr:write`, `pr:merge`, `pr:approve`
- `pr_comment:read`, `pr_diff:read`
- `status:read`, `status:write`
- `branch:list`, `branch:create`, `branch:push`
- `rate_limit:read`

### Branch Hierarchy

| Variable | Default | Description |
|----------|---------|-------------|
| `ADW_BRANCH_HIERARCHY` | `false` | Enable develop â†’ main promotion |
| `ADW_DEV_BRANCH` | `develop` | Integration branch name |
| `ADW_TARGET_BRANCH` | `main` | Default PR target branch |

### PR Scanning

| Variable | Default | Description |
|----------|---------|-------------|
| `ADW_ENABLE_PR_SCANNING` | `true` | Enable PR scanning in cron poller |

## Best Practices

1. **Use upstream routing for contributors**: Set `ADW_TARGET_REPO=upstream` when contributing to open source projects.

2. **Separate tokens**: Use different tokens for fork and upstream when possible for better security.

3. **Avoid pr:merge on automation tokens**: Require human review for merges.

4. **Enable branch hierarchy for team workflows**: Use `develop` as an integration branch for safer promotion to `main`.

5. **Keep tokens secure**: Never commit `.env` to version control.

## Next Steps After Configuration

1. Copy the generated snippet into your `.env` file
2. Replace placeholder tokens with actual values
3. Validate configuration:
   ```bash
   adw setup validate
   adw health
   ```
4. Test with a workflow:
   ```bash
   adw workflow plan <issue-number>
   ```
